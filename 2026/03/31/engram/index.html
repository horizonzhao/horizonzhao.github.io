Engram机制

Posted on 2026-03-31 In 技术分享
虽然MoE能减少计算，但对于静态知识（常识性知识、简单判断），有时仍需要大量计算，还是低效，没有把资源用在“刀刃”上，遂出现Engram...

虽然MoE能减少计算，但对于静态知识（常识性知识、简单判断），有时仍需要大量计算，还是低效，没有把资源用在“刀刃”上，遂出现Engram…

1. Engram架构

Deepseek在2026年1月12日发表论文《Conditional Memory via Scalable Lookup: A New Axis of Sparsity for Large Language Models》，提出了Engram架构，其思想与自然语言处理的N-gram有密切关系，下面进行分析。

原理：把海量的 N-gram 统计信息存储在一个巨大的查找表中，作为一个独立的模块挂载在大模型上。

作用：大模型负责复杂的推理，而 N-gram 模块负责快速提取固定的事实知识，Engram在结构上与MoE的条件计算形成互补。

优势：这相当于给大模型装了一个“超级字典”。当模型需要回答事实性问题时，直接查 N-gram 表，既不需要消耗昂贵的 GPU 算力，又能大幅减少幻觉。

1.1 传统N-gram存在的问题：

  1. 组合爆炸：比如3-gram，词库有50k个词，组合就有50k^3=1.25e14种可能，而真实遇到的词很稀疏

  2. 同一个词大小写不同，识别结果不同：Apple、apple、APPLE被当成不一样的，词库冗余

  3. 无法实现长距离依赖：窗口仅N个词

1.2 Engram解决方案：

  1. 分词压缩：Apple、apple、APPLE 都当成一个词，ID相同，压缩词库

  2. 多头哈希Multi-head Hashing

首先，需要理解哈希嵌入：不直接存储所有可能的N-gram，而是用哈希函数把N-gram映射到一个固定大小的空间。

如果我们要为“所有可能的 N-gram”都分配一个独立的存储空间，那这个空间是无限的。
Engram 的做法：设定一个上限，比如 100 万个槽位。

接着，问题来了，映射到有限空间，必然有两个不一样的词映射成相同值的情况。

于是，又提出多头哈希Multi-head Hashing：使用K个不同的哈希函数，每个都对应一个独立的嵌入表。最终把所有检索到的向量拼接起来。这样不同词重合的可能就大大降低了。

  1. 上下文感知门控

依赖Transformer，Engram 确保了大模型能看到上下文，并且在这个基础上，利用 N-gram 作为“索引”去查找和补充知识。

2. 具体例子

在工业级的大模型系统中，N-gram 常被用作混合架构的一部分，用来处理那些“杀鸡焉用牛刀”的简单任务。

2.1 智能体（Agent）的意图识别：

当用户说“我想订明天的机票”时，大模型处理这个请求成本很高。系统通常会先用轻量级的 N-gram（如 Trigram）进行快速匹配。如果命中了常见意图，直接用传统规则处理；只有遇到复杂的、没见过的指令，才调用大模型。这能极大降低延迟和成本。

2.2 大模型的“回退”机制：

类似于以前的输入法，现在的某些 AI 系统在生成文本时，如果遇到极低置信度的情况，或者需要保证特定的格式（如 JSON 字段名），会利用 N-gram 的概率分布进行约束或修正，确保输出的稳定性。
